# Value Betting Tracker — инструкции проекта

## О проекте
Это веб-приложение для ведения журнала ставок по системе value betting.

Приложение должно помогать пользователю:
- вручную добавлять ставки
- автоматически рассчитывать сумму ставки
- автоматически рассчитывать прибыль или убыток
- автоматически пересчитывать банк после каждой ставки
- хранить историю ставок
- сохранять данные после перезагрузки страницы

Это не просто таблица. Главная задача проекта — строго сохранять логику bankroll management и последовательный пересчёт банка.

## Главный приоритет
Точность бизнес-логики важнее визуального оформления.

Если возникает конфликт между красивым интерфейсом и правильным расчётом:
всегда сохраняй правильный расчёт.

## Обязательная структура таблицы
Главная таблица должна сохранять именно такую структуру колонок и именно в таком порядке:

| № | Дата | Событие | Ставка / Рынок | Коэфф | Перевес (%) | Выборка | Сумма | Результат | +/- | Банк после |

Не переименовывай эти колонки без прямого запроса пользователя.

## Начальный банк
Начальный банк по умолчанию: 500€.

Начальный банк должен быть редактируемым пользователем.
При изменении начального банка вся таблица должна полностью пересчитываться сверху вниз.

## Логика value betting

### Базовый процент ставки по перевесу
- если перевес < 3% → пропуск
- если перевес от 3% до менее 5% → 0.75%
- если перевес от 5% до менее 8% → 1%
- если перевес от 8% до менее 12% → 1.5%
- если перевес 12% и выше → 2%

### Множитель выборки
- 200 и больше → 1.0
- 100–199 → 0.9
- 50–99 → 0.8
- 30–49 → 0.7
- меньше 30 → пропуск

### Множитель коэффициента
- коэффициент ≤ 3.0 → 1.0
- коэффициент > 3.0 и ≤ 5.0 → 0.8
- коэффициент > 5.0 и ≤ 7.0 → 0.6
- коэффициент > 7.0 → пропуск

### Множитель типа рынка
- исходы → 1.0
- форы → 0.9
- тоталы → 0.9
- угловые / периоды → 0.9

### Множитель типа лиги
- топ-лиги → 1.0
- средние лиги → 0.9
- молодёжные / низкие лиги → 0.9

### Повышенный риск
WHL, MHL, junior и другие молодёжные лиги считаются повышенным риском.
Для таких ставок нужно дополнительно уменьшать сумму ставки через отдельный множитель риска, например 0.8.

## Формула расчёта ставки
Сумма ставки рассчитывается так:

Сумма ставки =
банк на момент ставки
× базовый процент по перевесу
× множитель выборки
× множитель коэффициента
× множитель рынка
× множитель лиги
× множитель риска, если применимо

После расчёта:
- округлять сумму до 0.5€
- если ставка валидна, можно применять минимальную сумму 1€, если это нужно логике приложения
- если ставка является пропуском, сумма должна быть 0

## Когда ставка считается пропуском
Ставка должна быть помечена как пропуск, если выполняется хотя бы одно из условий:
- перевес < 3%
- выборка < 30
- коэффициент > 7
- нестабильная лига
- подозрительный рынок

Ставки-пропуски можно сохранять в журнале, но:
- их сумма должна быть 0
- они должны быть явно визуально помечены
- они не должны обрабатываться как обычные активные ставки

## Классификация ставки
Каждая ставка должна получать классификацию:
- зелёная — сильная ставка
- жёлтая — нормальная ставка
- красная — рискованная ставка
- пропуск — не ставим

Классификация зависит от:
- величины перевеса
- размера выборки
- риска по коэффициенту
- типа лиги
- подозрительности рынка
- флага повышенного риска

## Допустимые результаты ставки
Поле результата должно поддерживать только такие значения:
- Не сыграно
- Выиграно
- Проиграно
- Возврат

## Логика прибыли и убытка
- Выиграно → сумма ставки × (коэфф - 1)
- Проиграно → -сумма ставки
- Возврат → 0
- Не сыграно → 0

## Правила пересчёта банка
Банк должен всегда пересчитываться строго сверху вниз по порядку строк.

Правила:
- первая строка использует текущий начальный банк
- каждая следующая строка использует банк после предыдущей строки
- изменение старой строки должно запускать полный пересчёт всех строк ниже
- удаление строки должно запускать полный пересчёт
- изменение порядка строк должно запускать полный пересчёт
- изменение результата должно запускать полный пересчёт
- изменение начального банка должно запускать полный пересчёт

## Поведение строк со статусом "Не сыграно"
Строки со статусом "Не сыграно" должны сохраняться в журнале.

Они не должны ломать последовательную логику банка.
Реализация должна сохранять корректный пересчёт банка сверху вниз.

## Правила целостности строк
- никогда не дублируй строки при редактировании
- если пользователь редактирует существующую ставку, нужно обновлять её, а не создавать новую
- нумерация должна оставаться корректной
- после удаления строки нумерация должна пересчитываться правильно

## Хранение данных
Используй localStorage для:
- начального банка
- списка ставок
- дополнительных настроек, если это полезно

После перезагрузки страницы данные должны сохраняться.
